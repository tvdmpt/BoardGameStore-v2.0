# BoardGameStore-v2.0

    Что было сделано
  1. Создана иерархия пользователей
Был создан абстрактный класс User, который содержит общие поля для всех пользователей: логин, пароль, email, имя. От него наследуются два класса: Employee для сотрудников и Customer для клиентов. Это позволяет хранить разные данные для разных ролей. Например, у сотрудника есть отдел и зарплата, а у клиента — баланс и предпочитаемый жанр игр.
  2. Реализован абстрактный метод
В классе User объявлен абстрактный метод performRoleSpecificAction. Он реализован по-разному в классах-наследниках. Сотрудник при входе выводит информацию о своем отделе. Клиент при входе выводит информацию о своих предпочтениях. Этот метод автоматически вызывается после успешной авторизации.
  3. Разные меню для разных ролей
Сотрудник видит меню с функциями управления: добавление, редактирование, удаление игр, импорт и экспорт данных. Администратор дополнительно видит пункты управления персоналом и статистики. Клиент видит только меню для покупок: просмотр каталога, поиск, корзина, история заказов, пополнение счета.
  4. Разная регистрация
Регистрация клиента доступна всем и требует минимум данных. Регистрация сотрудника защищена кодом доступа GAME2024. Без этого кода нельзя создать учетную запись сотрудника. При регистрации сотрудника указывается отдел, зарплата и наличие прав администратора.
  5. Разграничение прав в коде
В меню сотрудника проверяется флаг isAdmin. Если он false, пункты управления персоналом и статистики не отображаются. Это реализовано через простое условие if в методе showMenu.
  6. Полиморфизм в главном классе
В главном классе Main используется переменная типа User, которая может содержать объект Employee или Customer. В зависимости от фактического типа объекта вызывается соответствующее меню. Это позволяет избежать дублирования кода и легко добавлять новые роли.
    Где это используется
Абстрактный метод performRoleSpecificAction вызывается в методе login класса UserManager сразу после успешной проверки пароля. Это позволяет приветствовать пользователя персонализированным сообщением и выполнять ролевые действия автоматически.
Разные меню вызываются в главном цикле программы. Программа проверяет тип текущего пользователя через instanceof и вызывает соответствующий метод отображения меню.
    Преимущества подхода
Добавление новой роли требует только создания нового класса-наследника User и реализации трех абстрактных методов. Не нужно менять логику авторизации и основной цикл программы. Все ролевые проверки централизованы в соответствующих классах.
